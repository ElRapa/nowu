## Context

### Original Request
User identified that goal-layer v1 was too minimal: human-created goals, no UC mapping, no measurement infrastructure, only 3 goals. Goals need to serve as the "bigger picture" bridge — showing how the complete software connects, not just labeling epics. The vision calls for 90–99% AI work; goal creation should be agent-driven.

### Interview Summary
**Key Discussions**:
- Agent-created, human-reviewed: extend vision-bootstrap with sequential goal interview in same session
- UC↔goal bidirectional mapping: goals list UCs, USE_CASES.md gets parent_goal column
- Full metrics: success criteria + phase coverage table + UC completion %
- Solution Shape: outcome + capability map (which modules/capabilities, how they connect)
- Goal count: agent-determined from vision content
- Revert v1 commits, start clean
- idea-004: add altitude/phase frontmatter to goal template only (forward-compatible groundwork)
- Agent runtime: defer, keep Claude-native

**Research Findings**:
- vision-bootstrap has 5 interview questions, produces docs/vision.md [DRAFT]
- USE_CASES.md has 50 UCs in pipe-table: UC-ID | Title | stage_target | Primary Persona | Status
- Generated by use-case-agent@2.3 — parent_goal column requires agent update
- 3 Success Horizons in vision.md (6mo, 12mo, 24mo) are natural goal anchors

### Metis Review
**Identified Gaps** (addressed):
- Goal lifecycle / status transitions → template includes full lifecycle (proposed → in_delivery → achieved → retired)
- UC↔goal cardinality → many-to-many (UC can map to multiple goals)
- Bootstrap ordering → goals created first with empty UC lists, use-case-agent fills both sides later
- Existing 50 UCs backfill → NOT in scope (deferred, health-goals will flag as YELLOW not RED)
- Re-running vision-bootstrap → policy: regenerate goals, human reviews/merges with existing
- Goal count cap → soft cap at 8; agent flags if more, human decides
